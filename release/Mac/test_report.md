# Code-Sign & Notarization Report — BEAST.app

---

## v1 (`test-codesign.sh` before fix)

### Logic

```
1. find all files under Contents/ (excluding Contents/MacOS/* and *.dylib and *.so)
   → sign each with --options runtime --timestamp
2. sign outer BEAST.app bundle with --entitlements
3. verify
```

### Problems

| # | Problem | Notarytool error |
|---|---------|-----------------|
| 1 | **All `.dylib` and `.so` files were explicitly excluded** from the inner signing loop. The JRE contains ~40 dylibs (`lib/*.dylib`, `lib/server/libjvm.dylib`, `runtime/Contents/MacOS/libjli.dylib`, etc.) — none were signed with the Developer ID. | `"The signature of the binary is invalid."` |
| 2 | **No secure timestamp** on unsigned JRE binaries. Binaries left with the JDK vendor's original signature may lack a countersigned timestamp. | `"The signature does not include a secure timestamp."` |
| 3 | **`Contents/runtime` was never signed as a nested bundle.** It contains its own `Info.plist`, making it a nested bundle that Apple expects to be signed and sealed independently before the outer bundle is signed. | Outer bundle's `CodeResources` seals over an improperly signed nested bundle. |

## v2 (`test-codesign.sh` after fix)

### Logic

```
1. Sign ALL *.dylib and *.so files under Contents/
   → --force --options runtime --timestamp (no suppression, errors visible)
2. Sign all other non-dylib files (JARs, java, jspawnhelper, shaders.metallib)
   → excluding Contents/MacOS/* (signed as part of the outer bundle)
3. Sign Contents/runtime AS A BUNDLE
   → --force --options runtime --timestamp
4. Sign outer BEAST.app bundle last
   → --force --options runtime --timestamp --entitlements
5. Verify with --deep --verbose=2
```

This follows Apple's required **inside-out** signing order: deepest native binaries first, nested bundles next, outer bundle last.

## Remaining Problem (from notarytool log)

```json
{
  "severity": "error",
  "path": "BEAST.zip/output/BEAST.app/Contents/MacOS/BEAST",
  "message": "The signature of the binary is invalid.",
  "architecture": "arm64"
}
```

### What it means

`Contents/MacOS/BEAST` is the **native launcher binary** generated by `jpackage`. It is the `CFBundleExecutable` of the outer bundle — the one binary Apple scrutinises most strictly.

### Likely causes

1. **`MacOS/BEAST` must be signed explicitly before the outer bundle is signed.**
   When `codesign` signs a bundle it seals the main executable into `CodeResources`, but if the binary already carries a stale or mismatched signature from a previous `codesign` run (or from `jpackage`), the reseal can produce an invalid result. The fix is to sign `MacOS/BEAST` individually (with `--options runtime --timestamp`) **before** signing the outer bundle.

2. **The zip path includes `output/` as a prefix** (`BEAST.zip/output/BEAST.app/...` instead of `BEAST.zip/BEAST.app/...`). This does not cause an invalid signature but means the zip was created from the wrong working directory (`release/Mac/` instead of `release/Mac/output/`). Apple's path in the error reflects the zip structure, not necessarily a signing failure from a different location, but it should be corrected.

### Suggested fix

Add an explicit sign of the main executable before step 4:

```bash
# Between step 2 and step 3 (runtime bundle sign):
codesign --force --sign "$CODESIGN_IDENTITY" \
         --options runtime --timestamp \
         "$app/Contents/MacOS/BEAST"
```

And create the zip from inside `output/`:

```bash
cd "$OUTPUT" && zip -ry ../BEAST.zip BEAST.app/
```

## v3 (investigation after applying v2 fix)

### What was confirmed locally

After applying the v2 fix, local verification passed:

- `codesign --verify --deep` → **valid on disk**
- Entitlements correctly embedded in `MacOS/BEAST` (allow-jit, allow-unsigned-executable-memory, allow-dyld-environment-variables)
- Valid Developer ID chain + secure timestamp
- `spctl --assess` → **"Unnotarized Developer ID"** (expected — no ticket stapled yet, but signature is recognised as valid)

### Notarytool still rejected `MacOS/BEAST`

```json
{
  "severity": "error",
  "path": "BEAST.zip/BEAST.app/Contents/MacOS/BEAST",
  "message": "The signature of the binary is invalid.",
  "architecture": "arm64"
}
```

(Note: the `output/` path prefix from v2 is gone — zip path was corrected.)

### Root cause analysis

| # | Cause | Explanation |
|---|-------|-------------|
| 1 | **Wrong zip tool** *(most likely)* | `zip -ry` can silently drop macOS-specific metadata that codesign relies on to seal an app bundle. Apple's own documentation requires `ditto` for notarization submissions. `codesign --verify` passes locally because signatures are embedded in the Mach-O binary, but notarytool's stricter extraction may see a broken seal. |
| 2 | **Double bundle-level signing on `MacOS/BEAST`** | Signing `MacOS/BEAST` explicitly (v2 step 3) causes codesign to treat it as a full bundle signing — confirmed by `Format=app bundle` in `codesign -dvvv`. The outer bundle signing (step 5) then re-signs it a second time. Two consecutive bundle-level signings can leave `_CodeSignature/CodeResources` in a state that notarytool's stricter checker rejects, even though `codesign --verify` passes. |
| 3 | **Linter reordered steps 3 and 4** | The runtime bundle ended up being signed before `MacOS/BEAST`, reversing the intended order. Minor concern, not the root cause, but increases risk of the double-signing issue above. |

### Suggested fixes

**Fix 1 — replace `zip` with `ditto` (address cause 1):**

```bash
ditto -c -k --keepParent BEAST.app BEAST.zip
```

**Fix 2 — remove the explicit `MacOS/BEAST` signing step (address cause 2):**

The outer bundle signing (`codesign ... --entitlements ... BEAST.app`) already re-signs `MacOS/BEAST` with entitlements as the `CFBundleExecutable`. The extra explicit step creates an intermediate bundle-level signature that may interfere. Removing it simplifies the flow to:

```
1. Sign all dylibs / .so files
2. Sign other non-dylib files (java, jspawnhelper, JARs, etc.)
3. Sign Contents/runtime as a nested bundle
4. Sign outer BEAST.app with --entitlements
5. Verify
6. ditto -c -k --keepParent BEAST.app BEAST.zip
```

## v4 (Option C applied)

### Changes from v3

| # | Change | Reason |
|---|--------|--------|
| 1 | **Removed** explicit `codesign ... MacOS/BEAST` re-sign step | That step caused double bundle-level signing — the outer bundle signing always re-signs the main executable, so signing it first created an intermediate signature that notarytool rejected even though local `codesign --verify` passed. |
| 2 | **Added** `codesign --remove-signature` for all files under `Contents/MacOS/` | Strips any vendor-embedded (jpackage) signature load commands entirely before the outer bundle seals the binary. `--force` alone overwrites but may inherit corrupt metadata; `--remove-signature` guarantees a clean Mach-O. |
| 3 | **Generalised** to all launchers in `MacOS/` | Uses `find "$app/Contents/MacOS" -type f` instead of hardcoding `MacOS/BEAST`, so it handles any app bundle name (e.g. DensiTree). |

### Resulting signing order

```
1. Sign all dylibs / .so files (JRE native libraries)
2. Sign other non-dylib files (JARs, java, jspawnhelper, etc.) excluding MacOS/
3. Sign Contents/runtime as a nested bundle
4. Strip vendor signatures from all Contents/MacOS/ launchers (--remove-signature)
5. Sign outer BEAST.app with --entitlements (signs MacOS/ launchers once, cleanly)
6. Verify with --deep --verbose=2
```

### Why this is more robust than Option A

Option A (simply removing the explicit sign step) relies on `--force` being sufficient to override a potentially malformed jpackage signature. `--remove-signature` fully removes the Mach-O signature load command before the outer bundle seals it, eliminating any inherited corrupt state. The outer bundle then signs each launcher exactly once from a clean slate with the correct entitlements.

## v5 — jpackage `--mac-sign` approach (`test-jpackage-sign.sh`)

### Core change

Instead of post-hoc `codesign` on a pre-built app-image, use `jpackage --mac-sign` to sign the bundle **during creation**. This eliminates all dylib enumeration fragility and the double-signing problem entirely.

### Signing order

```
1. Stage JARs from Maven target (beast-fx/target/ and target/lib/)
2. jpackage --mac-sign --mac-entitlements  ← signs JRE dylibs, runtime bundle,
                                              MacOS/BEAST, and outer bundle atomically
3. Post-process BEAST.cfg (module-path switch), copy fxtemplates/
4. Restore bin/java (jpackage strips it); sign it individually
5. Strip xattrs from added resource files (com.apple.provenance cannot be removed
   by xattr -cr — see version.xml issue below)
6. Re-seal Contents/runtime (java binary added inside it)
7. Re-seal outer BEAST.app with --entitlements (runtime changed + resources changed)
   → re-signs MacOS/BEAST exactly once, cleanly — no double-signing
8. Verify: codesign --verify --deep + spctl --assess
9. Package: ditto -c -k --keepParent  (replaces zip, preserves macOS metadata)
```

### Result

```
valid on disk
satisfies its Designated Requirement
rejected — source=Unnotarized Developer ID   ← expected before stapling
```

Local signing passes. Ready for `xcrun notarytool submit`.

### version.xml root cause (discovered during v5)

#### Problem

`build-dmg.sh` copies `version.xml` to `Contents/version.xml` after jpackage runs.
Modern codesign `rules2` contains:

```
"^[^/]+$" => { "nested": true, "weight": 10 }
```

This rule matches **every plain file placed directly in `Contents/`** and requires it to be a signed nested code object (a bundle or Mach-O). A plain XML file can never satisfy this — codesign fails with:

```
code object is not signed at all
In subcomponent: .../BEAST.app/Contents/version.xml
```

This error persists regardless of xattr stripping because `com.apple.provenance` is a macOS-protected xattr that cannot be removed by user-space tools. However, xattrs are not the root cause — the `^[^/]+$` nested rule is.

#### Why it was not caught before

`build-dmg.sh` had no signing step. The `Contents/version.xml` placement was always incompatible with a signed bundle — signing is what exposed it.

#### Fix applied in test-jpackage-sign.sh

Removed the `Contents/version.xml` copy. `version.xml` is already present at `Contents/app/version.xml` (placed there by jpackage from staging).

#### Required follow-up (Java code change)

`BEASTClassLoader.initServices` locates `version.xml` by scanning the grandparent of each JAR. JARs are in `Contents/app/`, so it looks in `Contents/` — which can no longer hold the file. `BEASTClassLoader` must be updated to also check `Contents/app/version.xml`.

## Conclusion

Getting a jpackage-built BEAST.app through Apple's notarization is harder than it looks. We went through five iterations, each revealing a different layer of the problem. Here is a plain-English summary of everything we found and where things stand.

---

### 1. The signing problems (v1 → v4)

**v1** did not sign JRE dylibs at all and never treated the bundled Java runtime as a nested bundle. Apple rejected it immediately.

**v2** fixed those two issues but accidentally introduced a new one: `MacOS/BEAST` (the app launcher) was signed explicitly as a standalone binary, then signed again when the outer bundle was sealed. This double-signing left Apple's notarization service seeing an inconsistent result even though local verification passed.

**v3** identified the double-signing as the root cause. The fix was to strip the launcher's vendor-embedded signature cleanly before the outer bundle seals it, rather than trying to re-sign it explicitly.

**v4** applied that fix generically across all launchers in `Contents/MacOS/`, and replaced `zip` with `ditto` for packaging (zip silently discards macOS metadata that codesign needs).

---

### 2. The better approach (v5)

Instead of building the app first and signing it afterwards, we switched to using jpackage's built-in `--mac-sign` option. jpackage signs everything — JRE dylibs, the runtime bundle, the launcher, the outer bundle — in one atomic step during construction, in exactly the right order. This eliminates the entire class of "signing an already-built bundle" problems.

The only manual signing left to do is for `bin/java`, a binary that we add back into the bundle after jpackage runs (jpackage strips it). We sign that binary individually, then re-seal the two affected bundles (runtime and outer) to incorporate it. Local signing now passes cleanly.

---

### 3. The version.xml problem

During v5 we hit one more issue. The original `build-dmg.sh` copies `version.xml` into `Contents/version.xml` after jpackage has finished. This turned out to be incompatible with a signed bundle, for a non-obvious reason.

#### What the rule actually means

Apple's codesign embeds a set of built-in rules into every signed app bundle. One of those rules covers files placed directly inside `Contents/` (not inside any subfolder):

```
"^[^/]+$" => { "nested": true, "weight": 10 }
```

In plain English: **any file sitting directly in `Contents/` must be a signed bundle or executable** — something like a `.framework`, a nested `.app`, or a Mach-O binary. A plain XML file can never satisfy that requirement.

The only files Apple explicitly exempts from this rule are a small set of known names — each has its own higher-priority rule that overrides the nested check:

| File | Exempted by rule |
|---|---|
| `Info.plist` | `^Info\.plist$` |
| `PkgInfo` | `^PkgInfo$` |
| `version.plist` | `^version\.plist$` |
| `embedded.provisionprofile` | `^embedded\.provisionprofile$` |

`version.xml` matches none of these, so it falls through to the nested rule and codesign fails with:

```
code object is not signed at all
In subcomponent: .../BEAST.app/Contents/version.xml
```

#### Why it was not caught before

`build-dmg.sh` had no signing step. The `Contents/version.xml` placement was always incompatible with a signed bundle — signing is what exposed it. Stripping extended attributes (e.g. `com.apple.provenance`) does not help: the xattr is not the cause, and it is a macOS-protected xattr that cannot be removed by user-space tools anyway.

#### Why pre-placing it via jpackage still fails

A natural workaround is to use jpackage's `--resource-dir` option to put `version.xml` into `Contents/` before jpackage signs, so it is already there when codesign runs. This does not work either.

jpackage's `--mac-sign` calls Apple's standard `codesign` tool internally using the same built-in rules. If `version.xml` is in `Contents/` at sign time, codesign applies the `^[^/]+$` nested rule and fails for exactly the same reason — the error just moves from our re-seal step to jpackage's internal sign step. **There is no packaging trick that makes a plain file at `Contents/version.xml` acceptable under modern macOS codesign rules.**

#### The actual fix

The file is not missing. It already lives at `Contents/app/version.xml`, placed there by jpackage from the staging directory (staging contents go to `Contents/app/`). The only thing that needs to change is on the Java side:

**`BEASTClassLoader` must be updated to look for `version.xml` in the same directory as the JARs (`Contents/app/`) rather than one level up (`Contents/`).**
